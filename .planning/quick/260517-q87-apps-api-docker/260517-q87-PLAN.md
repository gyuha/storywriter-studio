## 현황 분석

### 현재 파일 배치

```
storywriter-studio/          ← 모노레포 루트
├── Taskfile.yml             ← 루트 태스크 (api:*, web:* 위임)
├── apps/
│   ├── api/                 ← Python FastAPI 앱
│   │   ├── Dockerfile            ← 멀티스테이지 프로덕션 빌드
│   │   ├── docker-compose.yml    ← 로컬 개발 인프라 (postgres, redis, mailpit)
│   │   ├── docker-compose.prod.yml  ← 프로덕션 오버레이
│   │   ├── .dockerignore
│   │   ├── pyproject.toml
│   │   ├── src/
│   │   └── alembic/
│   └── web/                 ← 프론트엔드 (React 계열 추정)
└── README.md
```

### Dockerfile 빌드 컨텍스트 분석

현재 `Dockerfile`의 `COPY` 명령은 모두 `apps/api/` 기준 상대 경로를 참조한다.

```dockerfile
COPY pyproject.toml uv.lock* ./
COPY README.md ./
COPY src/ ./src/
COPY alembic/ ./alembic/
COPY alembic.ini ./
```

`docker-compose.prod.yml`의 빌드 컨텍스트:

```yaml
build:
  context: .          # apps/api/ 디렉터리가 빌드 루트
  dockerfile: Dockerfile
```

즉 **빌드 컨텍스트는 `apps/api/`**이고, Dockerfile은 그 경계 안에서 완결된다.
루트 레벨 파일(`Taskfile.yml`, `apps/web/` 등)은 빌드에 관여하지 않는다.

### docker-compose 역할 분석

| 파일 | 역할 | 기동 서비스 |
|------|------|------------|
| `docker-compose.yml` | 로컬 개발 인프라 | postgres, redis, mailpit (FastAPI는 호스트에서 직접 실행) |
| `docker-compose.prod.yml` | 프로덕션 오버레이 | postgres, redis, app (mailpit 제외) |

두 파일 모두 **API 애플리케이션 전용 인프라**만 정의한다.
`apps/web/`을 위한 서비스(예: nginx, CDN 오리진)는 포함되지 않는다.

## 검토 결론

### 현재 위치(`apps/api/`) — 유지 권고

현재 위치가 **올바르다.** 이동 불필요.

근거는 아래 세 가지로 정리된다.

**1. 빌드 컨텍스트 경계가 정확히 일치한다**

Dockerfile이 `apps/api/` 외부 파일을 단 하나도 참조하지 않는다. 빌드 컨텍스트를
루트로 올리면 `apps/web/`, `node_modules/`, `.git/` 등 불필요한 대용량 디렉터리가
빌드 컨텍스트에 포함되어 이미지 빌드 속도가 저하되고 `.dockerignore` 복잡도가 증가한다.

**2. 책임 분리가 명확하다**

`apps/api/docker-compose.yml`은 API가 의존하는 인프라(postgres, redis, mailpit)만
정의한다. 모노레포 루트에 두면 "어느 앱의 인프라인가?"가 불분명해지고, 향후 `apps/web/`
이 별도 인프라를 추가할 때 파일 충돌 또는 네이밍 혼란이 발생한다.

**3. 루트 Taskfile이 이미 적절히 위임하고 있다**

```yaml
# 루트 Taskfile.yml
includes:
  api:
    taskfile: apps/api/Taskfile.yml
    dir: apps/api
```

`task api:infra`, `task api:test` 등 모든 API 명령은 루트에서 `task` 하나로 실행
가능하다. Docker 파일을 루트로 이동해도 DX(개발자 경험)가 개선되지 않는다.

## 대안 검토 (이동 시나리오)

이동을 고려할 만한 시나리오는 두 가지다. **현재 프로젝트에는 해당하지 않는다.**

### 시나리오 A: 루트 이동 (`/docker-compose.yml`)

**언제 적합한가?**

- 단일 앱 레포이거나, 여러 앱이 하나의 compose 파일로 통합 관리될 때
- 웹 앱도 Docker로 서빙되어 `apps/api` + `apps/web`이 하나의 compose에 묶일 때

**이 프로젝트에 적합하지 않은 이유:**

- `apps/web`은 호스트에서 `pnpm dev`로 실행되므로 compose 통합 대상이 아니다
- `Dockerfile`의 빌드 컨텍스트를 루트로 변경하면 모든 `COPY` 경로를 수정해야 한다
- 빌드 컨텍스트 크기가 불필요하게 커진다

### 시나리오 B: 전용 `docker/` 디렉터리 (`/docker/api/`)

**언제 적합한가?**

- 여러 앱이 각각 Dockerfile을 가지며 중앙 집중 관리가 필요할 때
- Kubernetes 매니페스트, Helm 차트 등 컨테이너 관련 파일이 대거 추가될 때

**이 프로젝트에 적합하지 않은 이유:**

- 현재 컨테이너화 대상이 `apps/api` 하나뿐이다
- `docker/api/Dockerfile`로 분리하면 빌드 컨텍스트 경로 설정이 복잡해진다
  (`context: ../../apps/api`)
- 실익 없이 디렉터리 뎁스만 증가한다

## 권고 사항

### 위치: 현행 유지 (`apps/api/`)

Docker 파일을 `apps/api/` 밖으로 이동하지 않는다.

### 단기 개선 제안 (선택사항)

현재 구조에서 보완할 수 있는 사항 두 가지를 제안한다.

**제안 1: `.dockerignore` 위치 확인**

`.dockerignore`는 `Dockerfile`과 **같은 디렉터리**에 있어야 Docker 엔진이 자동으로
인식한다. 현재 `apps/api/.dockerignore`로 올바르게 배치되어 있다. 유지한다.

**제안 2: 루트 `docker-compose` 미러링 불필요 확인**

일부 팀은 루트에 `docker-compose.yml` 심볼릭 링크를 두어 `cd apps/api` 없이 실행하려
한다. 그러나 `task api:infra`(루트 Taskfile 위임)가 이미 이 편의성을 제공하므로
심볼릭 링크는 불필요하다.
